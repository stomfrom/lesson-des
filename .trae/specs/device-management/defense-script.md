尊敬的各位老师，大家好。我是本次项目的负责人XXX。下面由我来为大家介绍我们的项目——星维设备管理系统。

## 一、开场白（约1分钟）

各位老师好。我们小组本次的项目是"星维设备管理系统"，一个面向企业的设备全生命周期管理平台。下面我将从项目背景、系统架构、功能实现、技术亮点和测试结果五个方面进行汇报。

---

## 二、项目背景与目标（约1.5分钟）

在日常的企业运营中，设备管理是一项非常重要但又繁琐的工作。传统的方式往往依赖纸质记录或者Excel表格，存在数据分散、无法实时追踪、权限混乱等问题。

针对这些痛点，我们开发了这套设备管理系统。系统主要面向两大类用户：一是**系统管理员**，负责整体的用户和权限管理；二是**普通操作员**，负责日常的设备维护操作。

我们设定的核心目标是：**让设备管理变得清晰、可控、可追溯**。具体来说，要做到设备的增删改查一体化、状态自动流转、权限精细化管控。

---

## 三、技术选型与系统架构（约2分钟）

接下来我介绍一下系统的技术架构。

**前端方面**，我们采用了 **Vue 3** 框架，配合 **Vite 8** 构建工具，UI 组件库选用 **Element Plus**。状态管理使用 **Pinia**，路由管理使用 **Vue Router**。这三个组合在目前的 Vue 生态中是最主流的选择，性能好、社区活跃。

**后端方面**，采用 **Express 4** 框架，数据库使用 **MySQL 8.0**，通过 **mysql2** 驱动以连接池方式连接。认证方面使用 **JWT（JSON Web Token）** 做无状态认证，密码使用 **bcryptjs** 进行加盐哈希存储。

**安全方面**，我们集成了 **Helmet**（HTTP 安全头）、**CORS**（跨域限制）、**express-rate-limit**（请求限流），并且对请求体大小做了 10KB 的限制，防止大请求攻击。

**系统的整体架构**是前后端分离的 SPA 模式。前端运行在 5173 端口，通过 Vite 的代理将 API 请求转发到后端的 3000 端口。后端采用三层架构：**路由层 -> 控制器层 -> 数据模型层**，每一层各司其职。

## 四、功能模块详解（约4分钟）

系统主要分为四个功能模块：设备管理、用户与权限管理、数据统计看板和生命周期配置。

### 4.1 设备管理模块

这是系统的核心模块，实现了设备的完整 CRUD 操作。

**设备列表页**支持分页展示，每页可切换 10/20/50 条。用户可以通过**设备名模糊搜索**或者**按状态多选筛选**来快速定位目标设备。表格的所有列——包括 ID、设备名、型号、位置、状态、维保日期——都可以点击表头进行**排序**。当筛选条件变化时，分页会自动重置到第一页，这个细节可以提升用户的操作体验。

**新增和编辑设备**，我们使用了同一个表单组件（DeviceForm.vue），通过路由中是否携带 id 参数来区分是新增还是编辑模式。表单中设备名、型号、位置为必填字段，前端有即时验证提示，后端也有同样的校验逻辑，形成双重保障。

**设备详情查看**：点击任意一行设备记录，会弹出一个详情对话框，展示该设备的完整信息，包括维保日期、创建时间和更新时间，时间都会自动转换为本地时区显示。

**删除操作**：我们使用了 Element Plus 的 Popconfirm 组件进行二次确认，并且确认框中会显示设备的具体名称，比如"确定要删除设备「服务器A」吗？"，避免误删除。

**CSV 导出**：列表顶部有一个"导出 CSV"按钮，可以将当前页的设备数据导出为 CSV 文件。我们特别做了三点优化：一是加入了 BOM 头确保 Excel 能正确识别 UTF-8 编码的中文；二是对以 =、+、-、@ 开头的单元格加了转义，防止 CSV 公式注入攻击；三是文件名中的非法字符做了替换，确保在 Windows 系统上也能正常下载。

**维保提醒功能**：系统会自动判断设备的维保状态。如果一个设备上次维保距今超过 11 个月（具体月数可以在设备配置中调整），状态栏会显示一个橙色的"即将维保"标签。这个计算排除了已报废的设备，而且对于从未维保过的设备也不会错误触发。

### 4.2 用户与权限管理模块

**用户认证**方面，我们采用 JWT 来做无状态认证。用户登录后，后端签发一个有效期 7 天的 token，前端持久化到 localStorage。每次请求时，Axios 的请求拦截器会自动在请求头中带上这个 token。

**路由守卫**方面，我们在前端路由的 beforeEach 钩子中做了两层检查：第一层检查用户是否已登录，未登录则跳转到登录页并保留目标 URL，登录后自动跳回；第二层检查页面是否需要管理员权限，非管理员访问管理员页面会被重定向到设备列表。

**权限控制**方面，我们实现了 RBAC——基于角色的访问控制。系统中的资源"device"支持四种操作：查看、新增、编辑、删除。管理员默认拥有全部权限，不需要查表。普通用户的权限存储在 permissions 表中，每次请求时由中间件动态加载。我们使用了一个工厂模式的中间件 `requirePermission(resource, action)`，在每一条设备路由上独立配置所需权限，做到了精确到按钮级别的权限控制。

**用户管理页面**只对管理员开放。在这个页面上，管理员可以查看所有用户的信息，新增用户，以及最核心的功能——通过勾选 checkbox 来为普通用户分配四项操作的任意组合。权限的修改是实时生效的，数据通过事务写入数据库，保证了原子性。此外，管理员还可以删除普通用户，但系统会阻止删除自己和其他管理员账号。

### 4.3 数据统计看板

Dashboard 是用户登录后默认看到的页面。这里展示了五个统计卡片：设备总数（蓝色）、正常运行（绿色）、维保中（橙色）、已报废（灰色）和即将维保（红色）。右侧还有一个"最近添加的设备"表格和快捷操作按钮。这些数据来源于后端的三条聚合查询，在访问 Dashboard 时自动获取。

### 4.4 设备生命周期自动流转

这个功能是我们系统的一个亮点。传统的设备管理系统需要人工手动去变更设备状态，而我们实现了自动化。

具体来说，每台设备在创建或编辑时，都可以单独配置两个参数：**维保触发月数**和**报废触发月数**，默认值分别是 11 个月和 12 个月。

然后，当用户访问 Dashboard 时，系统会自动执行两条 UPDATE 语句：
- 对于状态为"正常"且超过配置月数未维保的设备，自动变为"维保中"
- 对于状态为"维保中"且超过配置月数没有变化的设备，自动变为"已报废"

这样，设备管理员只需要定期更新维保日期，状态流转完全由系统自动完成，大大减少了人工管理的成本。

## 五、关键技术实现（约1.5分钟）

这里我想重点介绍几个关键技术的实现细节。

**第一，SQL 注入防护。** 我们全部使用 mysql2 的预处理语句（参数化查询），所有用户输入都通过 `?` 占位符传入，从根本上杜绝了 SQL 注入。即使是 LIKE 模糊搜索中的 `%` 和 `_` 通配符，我们也做了转义处理。

**第二，权限批量设置的原子性。** 在权限管理模块，更新用户权限时先删除旧权限再插入新权限。为了防止中间状态的数据丢失，我们将这两步操作包裹在数据库事务中。一旦插入失败，自动回滚到删除前的状态。

**第三，请求取消机制。** 在设备列表页面，当用户快速切换筛选条件或翻页时，可能会同时发出多个请求，后返回的响应可能覆盖先返回的，造成数据错乱。我们使用 AbortController 来取消上一次未完成的请求，保证只有最新的请求生效。同时，错误处理中通过检查 `ERR_CANCELED` 来忽略被取消的请求，避免弹出错误的提示信息。

**第四，生命周期自动流转。** 与常见的全局配置不同，我们允许每台设备独立配置自己的触发月数，这样灵活性更高。例如，关键设备可以设置更短的维保周期，而普通设备可以使用默认值。

## 附录：可能的答辩问题准备

**Q1: 为什么选择 JWT 而不是 Session？**
A: JWT 是无状态的，后端不需要存储会话信息，便于水平扩展。JWT 的载荷中可以直接存储用户 ID 和角色，减少数据库查询。适合前后端分离的 RESTful API 架构。

**Q2: 你们的权限控制和传统的 RBAC 有什么区别？**
A: 传统 RBAC 通常是角色→权限的映射。我们在此基础上做得更细——权限的粒度精确到资源+操作的组合（如 device:create），并且管理员可以针对每个用户独立分配四项操作的任意组合，而不是只能选择固定角色。

**Q3: 前端排序怎么实现的？**
A: 我们使用的是客户端排序。当用户点击表头时，触发 @sort-change 事件，记录排序字段和顺序到一个计算属性中，通过 sortedDevices 对当前页数据进行排序。对于当前页的数据量（最多 100 条），客户端排序完全够用。如果以后数据量增大，可以改为服务端排序。

**Q4: 如何防止一个用户删除后 Token 还能用？**
A: 我们在 JWT 认证中间件中，每次请求都会查询数据库校验用户是否存在。如果用户已被删除，即使 JWT 还未过期，也会返回 401"用户已被删除"。这种做法虽然增加了一次数据库查询，但保证了安全性。
